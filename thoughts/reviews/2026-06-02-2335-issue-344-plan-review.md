# Plan Review: Issue #344

## Decision

PLAN_REVISE

The plan is close, but it is not implementation-ready. It correctly understands the broad issue scope and most affected runtime surfaces, but it omits at least one current public seed-bearing API that violates the issue rule, and it does not sufficiently resolve the fact that the architecture guidance it relies on is present only as dirty/untracked local documentation changes in the current worktree.

## Issue Summary

GitHub issue #344 requests a cross-cutting refactor to introduce Rust newtypes for deterministic generation seed domains so seeds cannot be accidentally swapped across material, placement, biome, object-identity, elevation, planet, and solar-system domains.

The owner clarification in the issue comments strengthens the scope beyond a few representative root types:

- Every deterministic generation seed must have an enforced domain type, not a bare `u64`.
- Derived sub-seeds and stored generation keys must remain typed.
- No struct field named `*_seed` should be bare `u64`.
- No function parameter representing a seed should be bare `u64`.
- Registries, caches, observation keys, and generated identifiers must not key seed-domain data by bare `u64`.
- Raw `u64` is limited to explicit boundaries such as config/save/debug serialization, seed utility internals, and local hashing/mixing code.

Covered well:

- The plan includes the owner-comment vocabulary for `MaterialSeed`, placement seeds, object identity, biome climate, elevation, and chunk-level keys.

Not fully covered:

This is a public function whose parameter semantically represents a seed and is used for material generation naming. Under the issue rule and `seed-domain-typing.md`, it should not remain `seed: u64` unless the plan gives a precise architecture-compliant boundary justification. The current plan neither changes it to `MaterialSeed` nor explicitly classifies it as an allowed raw-bit helper boundary.
- Architecture-document handling is incomplete because the plan treats the seed-domain architecture rule as already durable while the current worktree shows the relevant file and routing changes are not cleanly committed in the repository state.

## Architecture and Autonomy Compliance

The plan largely respects the seed-domain architecture rule and autonomy boundaries:

- It avoids inventing new public seed vocabulary beyond names documented in the issue comment and architecture shard.
- It includes stop-and-ask triggers for additional public seed/key domains, public seed traits, central seed module decisions, field/event renames, system parameter growth, cross-plugin dependency changes, serde compatibility ambiguity, and overlapping dirty worktree changes.
- It preserves material-seed semantics: `MaterialSeed` is treated as a procedural input, not a material type identifier.
- It avoids unrelated Journal/KnowledgeGraph architecture inversion.

Architecture concerns requiring revision:

1. The plan makes an architecture-relevant choice about `PlanetSurface.detail_seed` by reusing `ElevationSeed` to avoid inventing `ElevationDetailSeed`. This may be acceptable, but the plan should make the stop-and-ask boundary sharper: if implementation reveals that base elevation and detail elevation must be compile-time-separated, it must stop rather than silently weakening domain separation. This is less severe than the `procedural_name` omission because the plan at least acknowledges the choice.
2. The plan relies on `seed-domain-typing.md`, but that file is untracked in the current worktree. Since the issue clarification changed durable architecture guidance, implementation must not proceed with the durable rule captured only in a GitHub comment or an untracked local file. The plan must require the relevant architecture docs under `docs/bmad/planning-artifacts/architecture/` to be part of the committed state for this work, or explicitly require that they be committed by a prerequisite before code migration begins.

## Findings

- BLOCKER: `src/naming.rs::procedural_name(seed: u64)` is omitted from the implementation plan. The function is public, its parameter semantically represents a deterministic seed used for material procedural naming, and it is not an obvious seed utility/internal local raw-bit helper. Issue #344 and `seed-domain-typing.md` require that no function parameter representing a seed remain bare `u64`. The plan must specify whether this API becomes `procedural_name(seed: MaterialSeed)` or provide an explicit, architecture-compliant reason it remains a raw-bit boundary.

- BLOCKER: The plan relies on `docs/bmad/planning-artifacts/architecture/cross-cutting/seed-domain-typing.md` and related routing/doc updates as durable architecture guidance, but current `git status` shows `seed-domain-typing.md` is untracked and several architecture docs are modified. The plan currently says not to modify or bundle unrelated architecture-doc changes, but these doc changes are directly relevant to issue #344. The plan must require that the architecture documentation updates be committed as part of this issue or be present as a clean prerequisite before implementation. A GitHub issue comment plus untracked local docs is not sufficient durable architecture handling.

- MAJOR: The plan’s handling of `PlanetSurface.detail_seed` as another `ElevationSeed` is plausible but should be guarded more explicitly. If implementation finds that detail elevation is a distinct seed domain crossing stored/helper boundaries, the implementer must stop and ask rather than reusing `ElevationSeed` by convenience. This should be reflected in the stop-and-ask triggers or phase notes.

- MINOR: The plan lists many tests but does not explicitly name some current helper/test seed call sites found by search, such as `tests/carry_processing.rs`, `tests/scenarios/helpers.rs`, `src/interaction.rs` test helpers, and `src/heat.rs` test helpers. Generic “any module-local tests touched by compilation errors” may be enough, but adding these to the test-update sweep would reduce implementation friction.

## Required Revisions

Before implementation, revise the plan as follows:

1. Add a concrete `src/naming.rs` migration step.
   - Preferred likely shape: import/use `MaterialSeed` and change `procedural_name(seed: u64) -> String` to `procedural_name(seed: MaterialSeed) -> String`, unwrapping `.0` only inside the local hashing logic.
   - Update callers in `src/materials.rs`, `src/fabricator.rs` tests/re-exports if needed, `src/naming.rs` tests, and any other direct call sites.
   - If the plan instead keeps `procedural_name` raw, it must explicitly justify why this public function is an allowed raw-bit boundary under `seed-domain-typing.md`; otherwise it violates issue #344.

2. Resolve architecture-document state explicitly.
   - State whether `docs/bmad/planning-artifacts/architecture/cross-cutting/seed-domain-typing.md` plus routing/index/material/data-architecture links are part of issue #344 implementation and must be included deliberately, or whether they must already be committed before implementation begins.
   - Do not classify these architecture-doc changes as unrelated dirty worktree changes if the implementation relies on them.
   - Ensure the plan requires durable architecture docs under `docs/bmad/planning-artifacts/architecture/`, not only GitHub issue comments, for the owner clarification.

3. Strengthen the `PlanetSurface.detail_seed` autonomy guard.
   - Either explicitly justify reusing `ElevationSeed` as same-domain elevation-detail data, or add a stop-and-ask trigger if detail elevation requires its own public seed domain type.

4. Expand the test/call-site update sweep to include direct current helper/test construction sites discovered by search, especially:
   - `tests/carry_processing.rs`
   - `tests/scenarios/helpers.rs`
   - `src/interaction.rs` test helpers
   - `src/heat.rs` test helpers
   - `src/naming.rs` tests

## Final Recommendation

Revise the plan before coding. The overall strategy and sequencing are solid, but implementation should not begin until the omitted public naming seed API and the architecture-doc worktree state are resolved concretely. Once those revisions are made, the plan will likely be ready for implementation.
