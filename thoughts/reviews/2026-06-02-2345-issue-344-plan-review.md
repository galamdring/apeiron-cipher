## Issue Summary

GitHub issue #344 requests compile-time seed-domain separation for deterministic generation inputs. The original issue names representative domains such as `MaterialSeed`, placement/biome/object identity seeds, and existing solar-system/planet seed types.

The owner clarification in the issue comments makes the rule stronger and architecture-wide:

- Every value that semantically represents a deterministic generation seed must have an enforced Rust domain type.
- Derived sub-seeds and stored generation keys must remain typed.
- No struct field named `*_seed` should be a bare `u64`.
- No function parameter representing a seed should be a bare `u64`.
- Registries, caches, observation keys, and generated identifiers must not key seed-domain data by bare `u64`.
- Raw `u64` values are limited to explicit boundaries: config/save/debug serialization, seed utility internals, and local hashing/mixing code where the bits are actually combined.

The issue comment also states that this rule has been added to `docs/bmad/planning-artifacts/architecture/cross-cutting/seed-domain-typing.md` and linked from world-generation/materials routing docs.
