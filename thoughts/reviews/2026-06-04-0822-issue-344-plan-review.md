# Plan Review: Issue #344

## Decision

PLAN_REVISE

The plan is close on requirements coverage and correctly targets the remaining known seed-domain typing gap, but it is not implementation-ready because it treats substantial uncommitted working-tree changes as if they are already part of `develop` and does not provide safe instructions for handling them. It also relies on architecture guidance that is currently present as untracked/modified architecture files in this workspace, without requiring those durable architecture documentation changes to be included or verified before implementation.

## Issue Summary

GitHub issue #344 requests a cross-cutting refactor to introduce Rust newtypes for deterministic seed domains so different generation domains cannot be accidentally interchanged as bare `u64` values. The owner clarification strengthens the requirement:

- Every deterministic generation seed value should have an enforced domain type.
- Derived sub-seeds and stored generation keys must remain typed.
- Struct fields named `*_seed`, semantic seed parameters, registries/caches/observation keys, and generated identifiers should not use bare `u64` outside explicit boundaries.
- Raw `u64` is allowed only at config/save/debug serialization boundaries, seed utility internals, and local hashing/mixing code.

The issue explicitly names affected areas including materials, observation keys, world generation profiles/sub-seeds, solar-system/planet seed types, and derivation call sites.

## Verification Assessment

The proposed verification is generally appropriate:

- Targeted grep checks for the known observation gap.
- `cargo fmt --check`, `cargo clippy --all-targets --all-features -- -D warnings`, `cargo test`, and `make check`.
- Focused raw-seed audit with manual classification of remaining seed-like `u64` occurrences.
- Determinism review to ensure no seed-mixing formulas or outputs changed.
- `git diff --check` and checks for newly introduced `pub(crate)`, `unsafe`, and `.unwrap()` violations.

One improvement is needed: verification should include an explicit `git status --short --branch` review before implementation and before final commit, with instructions not to include unrelated dirty files and to ensure architecture docs required by issue #344 are tracked or already landed.

## Findings

- **BLOCKER:** The plan’s current-state description is inaccurate/unsafe for this workspace. It says most issue #344 seed-domain typing work is already in current `develop`, but `git status` shows many relevant source and architecture files are modified or untracked. The plan does not instruct `rpi-implement` how to handle this dirty state, which creates a real risk of committing unrelated work or relying on changes that are not actually present in a clean branch.

- **BLOCKER:** The plan relies on durable architecture guidance in `docs/bmad/planning-artifacts/architecture/cross-cutting/seed-domain-typing.md`, but that file is currently untracked, with related architecture docs modified. Because issue #344’s clarification is architecture-impacting, the plan must require those architecture documentation updates to be tracked/included or explicitly verified as already landed before implementation completes. An issue comment alone is not sufficient durable architecture guidance.

- **MAJOR:** The plan’s workflow section should add a concrete pre-implementation state gate: inspect `git status --short --branch`, identify existing dirty changes, stop/ask if they are not owned by issue #344 or not explicitly approved to include, and avoid staging unrelated `.goose`, `.semantic-poc`, or other non-issue files.

- **MINOR:** Phase 2 uses “consider renaming” for private raw-bit helper parameters. This is acceptable as optional cleanup, but the plan should make the required outcome explicit: every remaining raw seed-like `u64` helper must either be clearly a local raw-bit boundary or be converted/clarified.

## Required Revisions

Before implementation, revise the plan to:

1. Add an explicit repository-state section based on the current workspace:
   - Note that `develop` is ahead of `origin/develop` and the working tree is dirty.
   - List the relevant modified/untracked issue #344 files or instruct the implementer to run `git status --short --branch` and classify them before changing code.
   - Distinguish committed `develop` state from uncommitted working-tree state. Do not say changes are “already in develop” unless they are committed.

2. Add a pre-implementation gate:
   - If existing dirty changes are not confirmed to belong to issue #344, stop and ask.
   - Do not stage or commit unrelated dirty files.
   - If the implementation is intended to continue from the current dirty seed-domain changes, say so explicitly and include them in the issue #344 change set.
   - If implementation should start from clean `develop`, require resetting/stashing only with explicit human direction, not autonomously.

3. Add explicit architecture-documentation handling:
   - Require `docs/bmad/planning-artifacts/architecture/cross-cutting/seed-domain-typing.md` and its routing/model/data-architecture links to be tracked/included in the implementation if they are not already committed on the implementation baseline.
   - Require verification that the durable architecture docs, not only the GitHub issue comment, capture the issue #344 seed-domain typing rule.

4. Update verification to include:
   - `git status --short --branch` before implementation and before final handoff/commit.
   - A check that no unrelated dirty files are staged/committed.
   - A check that architecture-doc changes required for issue #344 are tracked or already present in committed history.

5. Tighten Phase 2 wording so the implementer knows the required result for raw-bit helper documentation/naming: remaining raw seed-like `u64` helper parameters must be classified as explicit raw-bit boundaries or converted; renaming can remain optional only when existing docs already make the boundary clear.

## Final Recommendation

Revise before implementation. The seed-domain technical plan is sound, but the implementation baseline and architecture-doc state are ambiguous enough to block safe execution. Once the plan explicitly handles the dirty working tree and ensures the seed-domain architecture docs are durable, it should be ready for `rpi-implement`.
