# Plan Review: Issue #344

## Decision

PLAN_REVISE

The plan is directionally strong and covers most of the actual seed-plumbing blast radius, but it is not ready for `rpi-implement` because it converts unresolved architectural choices into implementation instructions. In particular, it invents additional public seed/key vocabulary and leaves alternative implementation paths open for generic noise helpers. Under the repository autonomy rules, those choices need either explicit owner clarification or durable architecture documentation updates before coding begins.

## Issue Summary

GitHub issue #344 requests a cross-cutting refactor so deterministic generation seed domains are not represented as interchangeable bare `u64` values. The issue scope names materials, observation keys, world profile sub-seeds, solar-system/planet seeds, and all derivation call sites.

The owner comment strengthens the requirement beyond representative newtypes:

- every deterministic generation seed/domain key needs an enforced domain type;
- no struct field named `*_seed` should be a bare `u64` except explicit config/save/debug/raw-bit boundaries;
- no seed-like function parameter should be a bare `u64`;
- registries, caches, observation keys, generated identifiers, and derived sub-seeds must remain typed;
- raw `u64` is allowed only at explicit boundaries such as serialization/config/debug formatting, seed utility internals, and local hashing/mixing.

The same owner comment says this rule has been added to `docs/bmad/planning-artifacts/architecture/cross-cutting/seed-domain-typing.md` and linked from world-generation/materials routing docs.

## Branch Workflow Assessment

Repository guidance loaded from `AGENTS.md` and required docs. Current branch/worktree inspection showed:

- current branch: `develop`;
- `main` is an ancestor of `develop`, so the `main`-ahead workflow-break condition is not present;
- worktree is dirty, with modified recipe files, modified architecture docs, deleted `.semantic-poc/*` files, untracked `docs/bmad/planning-artifacts/architecture/cross-cutting/seed-domain-typing.md`, and untracked `thoughts/` files.

The plan does not directly contradict the `AGENTS.md` branch model, but it also does not give implementers a safe preflight for the currently dirty worktree. A wide refactor touching many modules should explicitly reference `AGENTS.md` as the branch workflow source and require checking/protecting pre-existing uncommitted work before source edits begin.

The plan’s repeated “pause for manual confirmation before proceeding” notes are also not aligned with the normal agent workflow unless a documented stop-and-ask condition is actually encountered. Verification pauses should be self-check gates; human blocking should be reserved for ambiguity, architecture changes, or failed verification.

## Requirements Coverage

The plan covers the primary issue requirements:

- material catalog keyed by `MaterialSeed` instead of `u64`;
- `ObsKey` changed to use `MaterialSeed`;
- `WorldProfile` placement/object/biome/elevation sub-seeds typed;
- `SolarSystemSeed` and `PlanetSeed` retained/centralized;
- derivation call sites migrated toward typed helpers;
- raw `u64` restricted to config/save/debug/raw-bit boundaries;
- observation, journal, knowledge graph, combination, fabricator, exterior generation, and biome palette call sites included.

However, the plan goes beyond the issue/comment-specified vocabulary and makes public API choices without a clarification step. The owner’s comment gives examples and a rule, but the repository autonomy docs still require stopping when public type names or architectural choices are not specified.

## Architecture and Autonomy Compliance

The plan respects several important architecture rules:

- deterministic formulas and channel constants are to remain unchanged;
- material seed is not treated as material type identity;
- raw numeric config/asset boundaries are converted to typed runtime values;
- public newtypes are planned with docs and serde transparency;
- no UI, gameplay, or scheduling behavior changes are introduced.

It does not fully comply with the autonomy rules because it instructs implementers to proceed with newly invented public vocabulary and reusable API shapes. Public seed/key type names, a central `src/seeds.rs` module, and a reusable `SeedBits` trait are durable architectural vocabulary/API decisions, not private helper details.

If those decisions are accepted, they should be recorded in the relevant architecture docs under `docs/bmad/planning-artifacts/architecture/`, not only in this implementation plan or issue comments.

## Implementation Readiness

The plan is detailed and mostly concrete about files, phases, and likely call sites. It is not implementation-ready because some critical choices remain as alternatives instead of instructions:

- `continuous_value_field_01` may be changed to accept `impl SeedBits`, or left as a documented raw-bit boundary;
- elevation octave handling may add typed octave wrappers or convert to raw at the helper boundary;
- `From<u64>` / `From<Type> for u64` implementations are left as “only if useful,” even though these are public API choices;
- several new public types are listed as authoritative without issue-level confirmation.

`rpi-implement` should not have to decide these during coding.

## Verification Assessment

The verification plan is generally strong:

- `make check` is required;
- focused module tests are identified;
- deterministic equivalence tests compare typed helpers against existing `mix_seed` formulas;
- serde transparency is called out;
- targeted `rg` searches and allowlist categories are included.

Weaknesses:

- some search expressions are intentionally broad enough to produce many false positives, so the plan should say the final artifact must include a reviewed allowlist summary rather than only “review every hit”;
- repeated manual-confirmation pauses should be replaced with concrete automated/self-verification gates;
- the dirty-worktree state means verification should include a pre/post diff sanity check so unrelated existing changes are not accidentally bundled or overwritten.

## Findings

- **BLOCKER:** The plan invents public seed/domain-key vocabulary beyond what issue #344, the owner comment, and current architecture docs explicitly name. Examples include `ElevationDetailSeed`, `ChunkCoordinateKey`, `BiomeTemperatureNoiseSeed`, `BiomeMoistureNoiseSeed`, `ElevationOctaveSeed`, `ElevationDetailOctaveSeed`, and `OrbitalLayoutSeed`. Some may be correct, but under the autonomy rules, new public architectural names must be specified by the issue/owner or clarified before implementation.

- **BLOCKER:** The plan introduces durable public API/architecture choices without requiring architecture documentation updates. A central `src/seeds.rs` domain-vocabulary module, compatibility re-exports, and a public `SeedBits` trait are reusable architecture decisions. If accepted, the relevant architecture docs, likely `cross-cutting/seed-domain-typing.md` and possibly project-structure/routing docs, must be updated for traceability before implementation proceeds.

- **BLOCKER:** Phase 4 leaves important implementation alternatives unresolved for exterior noise/elevation boundaries. The plan says to choose between a `SeedBits` trait and a documented raw-boundary `u64` helper, and similarly leaves octave seed wrapping as an either/or. This is not ready for implementation because the implementer would have to make an architectural API decision.

- **MAJOR:** The plan’s repeated “pause for manual confirmation before proceeding” notes conflict with a pipeline-style implementation flow unless a specific stop-and-ask condition is hit. These should become self-verification checkpoints, with blocking only for the explicit ambiguity/architecture cases.

- **MAJOR:** The plan does not account for the current dirty worktree. Because issue #344 is a wide refactor, the implementation plan should require an initial `git status` check and explicit protection of pre-existing unrelated modifications before editing source files.

- **MINOR:** The final raw-seed search strategy is useful but very broad. The plan should require a concise reviewed allowlist summary in the implementation notes/issue comment so reviewers can distinguish intentional raw boundaries from missed runtime seed plumbing.

## Required Revisions

1. Add an explicit clarification step before implementation for every public seed/key type name not already specified by issue #344, the owner comment, or `seed-domain-typing.md`. At minimum, resolve whether the following are approved public vocabulary or should remain private/local/raw-boundary details:
   - `ElevationDetailSeed`
   - `ChunkCoordinateKey`
   - `BiomeTemperatureNoiseSeed`
   - `BiomeMoistureNoiseSeed`
   - `ElevationOctaveSeed`
   - `ElevationDetailOctaveSeed`
   - `OrbitalLayoutSeed`
   - any `PlanetCountSeed` / `PlanetEnvironmentSeed` style additions if implementation pressure appears.

2. Resolve and document the shared API shape before coding:
   - whether `src/seeds.rs` is the approved central home for seed-domain vocabulary;
   - whether existing `world_generation::PlanetSeed` and `solar_system::SolarSystemSeed` should remain re-export compatibility paths;
   - whether a public `SeedBits` trait is approved, or whether generic noise helpers remain explicitly documented raw-bit utility boundaries;
   - whether `From<u64>` / `From<SeedType> for u64` should exist or whether `.new()` / `.raw()` are the only approved conversions.

3. If any of the choices above are accepted, require updates to the relevant architecture docs under `docs/bmad/planning-artifacts/architecture/` before implementation, not just an issue comment. `cross-cutting/seed-domain-typing.md` should capture durable vocabulary/API rules; routing/project-structure docs should be updated if the central module location is intended architecture.

4. Replace Phase 4 alternatives with one concrete implementation path. Do not leave `rpi-implement` to choose between `SeedBits` and raw-boundary `u64`, or between typed octave wrappers and local raw conversions.

5. Replace “pause for manual confirmation after each phase” with:
   - run the phase’s automated checks;
   - inspect targeted searches;
   - continue if clean;
   - stop only for the plan’s explicit stop-and-ask cases or failed verification that cannot be resolved without changing scope/architecture.

6. Add a preflight instruction that references `AGENTS.md` as the branch workflow source and requires checking current branch/worktree state before edits. The instruction should tell implementers to avoid overwriting or bundling unrelated pre-existing changes.

7. Strengthen final verification by requiring the implementation result to summarize remaining raw `u64` seed/search hits by allowlisted category: config/save/debug/asset schema boundary, channel constant/config, raw-bit utility internals, non-seed numeric values, and test literals before wrapping.

## Final Recommendation

Revise the plan before implementation. The core migration strategy is sound, but public vocabulary/API decisions and generic seed-helper architecture must be clarified and documented first. Once those decisions are made explicit and Phase 4 alternatives are collapsed into concrete instructions, the plan should be ready for a wide but mechanical implementation pass.
