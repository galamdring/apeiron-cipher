# Research: Apeiron Cipher GitHub issue #344 seed newtypes

## Research Question

Research GitHub issue #344 only: `refactor: introduce seed newtypes for domain separation (MaterialSeed, PlacementSeed, etc.)`.

Issue #344 describes that sub-seeds throughout the generation stack are currently bare `u64` values. Although `PlanetSeed` exists as a newtype, derived sub-seeds shed type safety and can be passed across domains. The issue scope names these areas:

- `src/materials.rs` — `MaterialCatalog` keyed by `MaterialSeed` instead of `u64`
- `src/observation.rs` — `ObsKey` uses `MaterialSeed`
- `src/world_generation.rs` — `WorldProfile` sub-seeds get proper types
- `src/solar_system.rs` — system/planet seed types
- All derivation call sites

## Summary

The current codebase already contains two seed newtypes: `SolarSystemSeed(pub u64)` in `src/solar_system.rs` and `PlanetSeed(pub u64)` in `src/world_generation.rs`. Most other seed domains in issue #344 remain plain `u64` values. The procedural generation stack uses `seed_util::mix_seed(base: u64, channel: u64) -> u64` and a `SeedChannel` enum to derive deterministic sub-seeds, then stores or passes those derived values as raw integers.

Material generation is currently keyed by raw `u64` material seeds: `GameMaterial.seed` is a `u64`, `MaterialCatalog.by_seed` is a `HashMap<u64, GameMaterial>`, and catalog APIs accept or return `u64`. Observation confidence tracking also keys by raw material seed through `type ObsKey = (u64, PropertyName)`. World generation derives placement, object identity, biome climate, and elevation sub-seeds from `PlanetSeed`, but stores those sub-seeds as raw `u64` fields on `WorldProfile` and `ChunkGenerationKey`. Solar-system generation wraps the system root seed in `SolarSystemSeed` and planet outputs in `PlanetSeed`, but star, orbital-layout, planet-count, and planet-environment internal derivations use raw `u64` intermediate values.

The routed architecture docs state that material seeds are procedural generation inputs rather than type identifiers, registries grow on demand from seed-derived material facts, and deterministic generation depends on seeded derivation and explicit scheduling. The code currently reflects the seed-as-input model while retaining raw integer seed plumbing for most derived domains.

## Detailed Findings

### Seed utility primitives

- `src/seed_util.rs:29` defines `pub fn mix_seed(base: u64, channel: u64) -> u64`, the central deterministic seed-mixing primitive used by materials, world generation, and solar-system generation.
- `src/seed_util.rs:138` defines `SeedChannel::mix_seed(self, base: u64) -> u64`, a convenience method that still accepts and returns raw `u64`.
- `src/seed_util.rs:148` defines `seed_to_unit_f32(mixed: u64) -> f32`, which converts a mixed raw seed into a deterministic unit-interval float.
- `src/seed_util.rs` contains many named seed channel constants and the `SeedChannel` enum. The channel system separates derivation channels by constants, while the values being mixed remain raw integers.

How it connects:

- `materials.rs` imports `mix_seed` and material channel constants to derive material properties from one material seed.
- `world_generation.rs` imports `mix_seed` and world-generation channel constants to derive profile sub-seeds, chunk keys, biome noise seeds, elevation seeds, and generated IDs.
- `solar_system.rs` imports `mix_seed` and solar-system channel constants to derive star profiles, planet counts, orbital layouts, planet seeds, and planet environments.
- `world_generation/exterior.rs` uses raw seed keys and local mixing helpers for exterior mineral placement and material palette selection.

## Open Questions

- Which exact seed domains should receive public newtypes beyond the issue examples (`MaterialSeed`, `PlacementSeed`, `BiomeSeed`, `ObjectIdentitySeed`) is not exhaustively enumerated in issue #344.
- Whether root config fields such as `WorldGenerationConfig::solar_system_seed` and the configured planet seed should remain raw serialized `u64` values or use serializable newtypes is not specified in issue #344.
- Whether interior helper seeds such as solar-system `layout_seed`, elevation octave seeds, lattice/candidate seeds, and exterior variation keys should become domain newtypes or remain raw local derivation values is not specified in issue #344.
- Whether `WellKnownMaterial::seed()` should return a `MaterialSeed` or remain a raw compatibility boundary is not specified in issue #344.
